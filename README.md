# Provenance Guard

Provenance Guard is a backend service that creative sharing platforms can plug into when they need help classifying submitted text. It accepts creative writing or platform posts, analyzes the text with multiple detection signals, returns an attribution result with a confidence score, generates a reader-facing transparency label, and records both decisions and appeals in an audit log.

## Required Feature Map

| Requirement | Provenance Guard design |
| --- | --- |
| Content submission endpoint | `POST /submit` accepts text content and returns `content_id`, attribution result, confidence score, signal details, transparency label, and current status. |
| Multi-signal detection pipeline | The classifier uses at least two separate signals: Groq model attribution review and pure-Python stylometric heuristics. |
| Confidence scoring with uncertainty | Signals are combined into a normalized confidence score. Scores near the decision boundary produce an `uncertain` result and a softer label than high confidence decisions. |
| Transparency label | The API returns exact reader-facing text for high-confidence AI, high-confidence human, and uncertain classifications. |
| Appeals workflow | `POST /appeal` captures the creator's reasoning, links it to the original decision, logs the appeal, and changes the content status to `under_review`. |
| Rate limiting | `POST /submit` is rate limited to protect the service from spam, scraping, and accidental overload. |
| Audit log | Every decision and appeal is written to a structured audit log, including confidence score, signals used, label, and status changes. |

## Architecture Narrative

A single piece of text enters Provenance Guard through the **API Gateway / Flask App**, which exposes `POST /submit`. The endpoint validates that the request contains text, rejects empty or oversized submissions, checks the caller against the rate limit, and creates a new `content_id` so every later decision can be traced.

The accepted text moves into the **Submission Service**. This component stores the original submission metadata, sets the initial status to `classified_pending_log`, and sends only the text needed for classification into the detection pipeline. It is responsible for keeping the API layer thin and making sure every submission follows the same workflow.

The **Detection Pipeline** runs two independent signals. The first signal is **Groq model attribution review**, using `llama-3.3-70b-versatile` to judge whether the text reads as human-written or AI-generated by evaluating semantic and stylistic coherence holistically. This signal looks for patterns such as generic development, overly even structure, low lived specificity, and formulaic transitions. It is useful because a language model can evaluate higher-level writing behavior that is hard to capture with simple counts, but it cannot prove authorship and can be wrong when a human writes in a polished or formulaic style. The second signal is **stylometric heuristics**, implemented in pure Python, which measures structural statistics: sentence length variance, type-token ratio, punctuation density, average sentence complexity, and repetition. This signal is useful because human and AI writing often differ in measurable rhythm and variation, but it is weaker on very short texts, constrained genres, and heavily edited work. These signals are genuinely distinct: one is semantic and holistic, while the other is structural and statistical.

The signal outputs move to the **Confidence Scoring Engine**. This component combines the signal scores into one attribution score and then converts that into a confidence value. If the combined evidence is strong, it returns `likely_ai` or `likely_human` with high confidence. If the evidence is mixed or close to the decision boundary, it returns `uncertain`. A score like `0.51` should not be treated the same as `0.95`: `0.51` means the system barely leans one way and should show the uncertain label, while `0.95` means the evidence is strong enough for a high-confidence label.

The classification then moves to the **Transparency Label Builder**. This component turns the technical result into plain language that a reader can understand. It chooses one of the three approved label variants, inserts the confidence percentage, and returns that text in the API response alongside the structured classification fields.

Before the response leaves the system, the **Audit Logger** records the full decision. The log entry includes the `content_id`, timestamp, attribution result, confidence score, transparency label text, signal names, signal scores, and current status. This audit record is the canonical evidence for how the system reached the label.

The user-facing platform receives the response and displays the returned transparency label to readers. The platform does not need to understand the internal signal math; it only needs the label text, confidence score, result, and content status.

If a creator disagrees with the classification, the platform sends an appeal to **Appeals Service** through `POST /appeal`. The appeal includes the `content_id` and the creator's reasoning. The Appeals Service stores that reasoning, links it to the original audit decision, changes the content status to `under_review`, and writes a second audit event showing that the classification has been contested. Automated re-classification is not required for the first version; the appeal creates a review trail for a human moderator or future review process.

## API Contract

### `POST /submit`

Request:

```json
{
  "text": "A poem, short story excerpt, blog post, or other text content."
}
```

Response:

```json
{
  "content_id": "cnt_001",
  "status": "classified",
  "attribution": "likely_ai",
  "confidence": 0.95,
  "transparency_label": "Likely AI-generated. Provenance Guard found strong signs of AI-generated writing patterns. Confidence: 95%.",
  "signals": [
    {
      "name": "groq_model_attribution_review",
      "score": 0.88,
      "summary": "The model found polished structure, generic transitions, and low lived specificity."
    },
    {
      "name": "stylometric_heuristics",
      "score": 0.93,
      "summary": "Sentence length variance, vocabulary diversity, punctuation density, and repetition look unusually uniform."
    }
  ]
}
```

### `POST /appeal`

Request:

```json
{
  "content_id": "cnt_001",
  "creator_reasoning": "This was drafted from my personal journal and edited manually."
}
```

Response:

```json
{
  "appeal_id": "apl_001",
  "content_id": "cnt_001",
  "status": "under_review",
  "message": "Appeal received. The original classification is now under review."
}
```

### `GET /log`

Returns structured audit events for attribution decisions and appeals.

## Recommended Stack

| Component | Tool | Notes |
| --- | --- | --- |
| API framework | Flask | Free and lightweight. |
| Detection signal 1 | Groq with `llama-3.3-70b-versatile` | LLM-based classification that captures semantic and stylistic coherence holistically. |
| Detection signal 2 | Stylometric heuristics | Structural statistics in pure Python: sentence length variance, type-token ratio, punctuation density, average sentence complexity, and repetition. |
| Rate limiting | Flask-Limiter | Free and integrates with Flask. |
| Audit log | Structured JSON first, SQLite optional later | No additional setup required for the first version. |

## Detection Signals

| Signal | Property measured | Why this can differ between human and AI writing | Blind spots |
| --- | --- | --- | --- |
| `groq_model_attribution_review` | LLM-based classification of whether the text reads as human-written or AI-generated. It captures semantic and stylistic coherence holistically: generic development, overly balanced structure, low lived specificity, smooth but formulaic transitions, and lack of concrete authorial context. | Generated text often optimizes for fluent, complete, and broadly acceptable prose. That can create writing that feels polished but generic, evenly structured, and light on strange or personal detail. Human writing can be more uneven, idiosyncratic, specific, or messy in ways a model can recognize at the passage level. | It cannot know who actually wrote the text. It may over-flag polished human writing, classroom essays, marketing copy, or writers using a very conventional style. It may under-flag AI text that has been heavily edited or prompted to include personal detail. Its judgment can also vary with prompt wording and model behavior. |
| `stylometric_heuristics` | Structural statistics computed in pure Python: sentence length variance, type-token ratio for vocabulary diversity, punctuation density, average sentence complexity, repeated words or phrases, and paragraph regularity. | Human writing often contains irregular rhythm, uneven sentence lengths, surprising punctuation choices, and topic-specific vocabulary. AI-generated text can trend toward smoother pacing, repeated structures, balanced sentence lengths, and safer word choice, especially in short explanatory prose. | It does not understand meaning, intent, or authorship. Very short submissions do not provide enough measurements. Poems, dialogue, highly edited prose, templates, and intentionally minimalist writing can confuse it. Human writers can be formulaic, and AI writing can be edited until its rhythm looks human. |

The system must use both signals for every classification. A single signal is not enough to return a final attribution result. These two signals are genuinely independent because the Groq signal is semantic and holistic, while the stylometric signal is structural and statistical. Combining them is more informative than asking two versions of the same question.

## Confidence Scoring

Provenance Guard treats confidence as a measure of how strong the combined evidence is, not just which side wins. The detection pipeline produces signal scores from `0.0` to `1.0`, where higher values indicate stronger AI-like evidence. A `0.5` score means the system cannot responsibly lean AI or human from the available evidence.

The scoring design intentionally treats false positives as worse than false negatives. Labeling a human creator's work as AI-generated can damage trust and reputation, so `likely_ai` requires stronger evidence than a simple majority lean.

Planned thresholds:

| Combined score | Result | Label family |
| --- | --- | --- |
| `>= 0.80` | `likely_ai` | High-confidence AI |
| `<= 0.30` | `likely_human` | High-confidence human |
| `> 0.30` and `< 0.80` | `uncertain` | Uncertain |

Testing approach:

- Use clearly human-authored sample text, clearly AI-like sample text, and mixed/edited samples.
- Confirm high-confidence examples produce scores near the ends of the range.
- Confirm borderline examples produce uncertain labels instead of pretending to be certain.
- Confirm a `0.51` score produces the uncertain label while a `0.95` score produces the high-confidence AI label.

## False Positive Scenario

A false positive happens when a human writer submits original work and Provenance Guard classifies it as AI-generated. This is a serious product risk because the label could affect how readers judge the creator's honesty or effort.

Scenario trace:

1. The creator submits a polished essay, story excerpt, or poem through `POST /submit`.
2. The Groq model attribution review may flag the piece because it has smooth structure, formal transitions, and few personal details.
3. The stylometric heuristics may also flag it if the sentences are unusually even, punctuation is regular, vocabulary diversity is low, or paragraph lengths are balanced.
4. If both signals strongly point toward AI-like writing, the system may return `likely_ai` with high confidence. If the signals disagree or land near the middle, the system should return `uncertain` instead.
5. The transparency label must avoid calling the creator dishonest. It says `"Likely AI-generated..."`, not "This was definitely written by AI."
6. The audit log records the exact signal scores and summaries so the decision can be reviewed later.
7. The creator can submit `POST /appeal` with their reasoning, such as drafting history, source notes, or an explanation of their style.
8. The appeal is logged beside the original decision and the content status changes to `under_review`.

Design decision from this scenario: confidence must reflect uncertainty when evidence is mixed. A human writer with one AI-like signal and one human-like signal should receive an uncertain label, not a high-confidence AI label. High-confidence AI should require agreement across both distinct signals. The appeal path exists because even high-confidence classifications can be wrong.

Perfect AI detection is not the goal because perfect AI detection is not a solved problem. The system should acknowledge uncertainty honestly, avoid accusatory labels, preserve evidence in the audit log, and give creators a usable path to appeal.

## Transparency Label Text

The API returns one of these exact label variants. The confidence percentage is filled in from the decision score.

| Variant | Exact label text |
| --- | --- |
| High-confidence AI | `"Likely AI-generated. Provenance Guard found strong signs of AI-generated writing patterns. Confidence: {confidence_percent}%."` |
| High-confidence human | `"Likely human-written. Provenance Guard found strong signs of human writing patterns. Confidence: {confidence_percent}%."` |
| Uncertain | `"Origin uncertain. Provenance Guard found mixed signals, so this text should not be treated as clearly AI-generated or clearly human-written. Confidence: {confidence_percent}%."` |

## Appeals Workflow

Creators can contest a classification with `POST /appeal`. The appeal workflow:

1. Accepts `content_id` and `creator_reasoning`.
2. Finds the original attribution decision.
3. Stores the appeal with its own `appeal_id` and timestamp.
4. Updates the content status to `under_review`.
5. Writes an audit-log event linking the appeal to the original decision.

The appeal does not automatically change the attribution result. It creates a review trail so a moderator or later review process can evaluate the creator's reasoning.

## Rate Limiting

Chosen limit for `POST /submit`: **10 submissions per minute per client IP**.

Reasoning:

- A creative platform integration needs enough room for normal interactive use.
- Ten submissions per minute supports drafting, retries, and small batch checks.
- The limit is low enough to reduce spam, accidental loops, and expensive repeated analysis.
- Appeals are not expected to be high-volume, but they should still be protected separately in implementation.

Planned implementation: Flask-Limiter on `POST /submit`, using the remote address as the default key.

The number is not arbitrary: a normal creator may submit a piece, revise it, retry after validation feedback, or test a few drafts in a short session. An adversary, by contrast, may try to flood the endpoint to consume Groq calls, create noisy audit logs, or probe detector thresholds with many prompt variations. Ten per minute keeps normal interactive use comfortable while slowing obvious abuse.
